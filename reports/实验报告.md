# 气象大数据采集、分析与可视化系统实验报告

## 一、实验目标与工作约束

本课程设计实现一个气象大数据采集、存储、计算与可视化系统。系统固定展示唐山、北京、上海三个站点。唐山站用于硬件采集演示，温湿度来自 ESP32-C3 与 DHT11；北京、上海以及各站点的气压、风速、风向、天气代码来自 Open-Meteo 真实气象接口。后端保留 HDFS、HBase、Hive、Spark 处理流程，Web 服务使用 FastAPI，对外提供 JSON 接口并托管静态可视化页面。

本报告与实现过程同步维护。每完成一项环境配置、数据写入、计算任务或页面验证，就记录对应命令、结果和截图，避免在系统完成后补写实验过程。

## 三、数据来源与数据模型

### 3.1 Open-Meteo 真实气象数据

本项目使用 Open-Meteo Forecast API 获取当前气象数据，使用 Historical Weather API 获取历史气象序列。Open-Meteo 接口按经纬度返回气象字段，本系统使用字段如下：

| Open-Meteo 字段 | 系统字段 | 单位 |
|---|---|---|
| `temperature_2m` | `temperature` | 摄氏度 |
| `relative_humidity_2m` | `humidity` | % |
| `surface_pressure` | `pressure` | hPa |
| `wind_speed_10m` | `wind_speed` | m/s |
| `wind_direction_10m` | `wind_direction` | 度 |
| `weather_code` | `weather_code` | WMO 天气代码 |

三个站点的经纬度如下：

| 站点 | station_id | 纬度 | 经度 |
|---|---|---:|---:|
| 唐山 | tangshan | 39.63 | 118.18 |
| 北京 | beijing | 39.90 | 116.40 |
| 上海 | shanghai | 31.23 | 121.47 |

### 3.2 统一记录格式

系统内部统一使用以下字段：

```csv
collect_time,station_id,station_name,temperature,humidity,pressure,wind_speed,wind_direction,weather_code
```

其中 `collect_time` 使用 Asia/Shanghai 时区。写入 Hive 外部表和 Spark 分析时保留完整字段，前端接口再按页面需要返回当前值、趋势序列、统计摘要和记录表。

### 4.1 ESP32-C3 MicroPython 固件烧录

按照最终指导文档，ESP32-C3 需要先烧录 MicroPython 固件。本次使用 `数据采集.zip` 中的 `ESP32_GENERIC_C3-20250415-v1.25.0.bin`，设备通过 USB 枚举为 `/dev/ttyACM0`，稳定设备路径为 `$ESP32_PORT`。

当前主机用户不在 `uucp` 串口组中，因此需要对当前设备节点临时授予读写权限。刷写过程中 `esptool` 的 stub flasher 在该 ESP32-C3 上出现 `Illegal instruction`，因此改用 `--no-stub` 模式，并用 `erase-region 0x0 0x400000` 擦除 4MB flash 区域，再写入 MicroPython 固件。

```bash
./scripts/flash-esp32-firmware.sh
```

### 4.2 DHT11 传感器 GPIO4 采样验证

最终指导文档中的 DHT11 示例使用 IO02 读取数据。本次实物连接中，DHT11 数据线接入 ESP32-C3 的 GPIO4，因此测试程序改为：

```python
sensor = dht.DHT11(Pin(4, pull=Pin.PULL_UP))
```

测试程序连续采样 5 次，均能读出温度和湿度，说明 ESP32-C3、MicroPython 固件、DHT11 传感器和 GPIO4 接线可正常工作。

```bash
uvx mpremote connect $ESP32_PORT run esp32/dht11_gpio4_test.py
```

### 4.3 写入 GPIO4 版气象站程序

在传感器读数验证通过后，先尝试按最终指导文档使用 Wi-Fi 热点和 TCP 服务端传输数据。由于当前开发机只有一块无线网卡，直接开启热点会断开宿主机正在使用的上游 Wi-Fi；虚拟 AP 方案又受到当前 5GHz 上游信道和 ESP32-C3 仅支持 2.4GHz 的限制。因此本阶段改用 USB CDC 串口传输唐山站温湿度数据，先保证硬件采集数据可以稳定进入宿主机程序。

USB 串口发送格式如下：

```csv
station_id,station_name,temperature,humidity
```

示例：

```csv

```

写入 USB 版自动运行程序：

```bash
uvx mpremote connect $ESP32_PORT cp esp32/usb_weatherstation_main.py :main.py
```

## 五、系统运行说明

### 5.1 启动顺序

系统完整版的启动顺序如下：

1. 连接 ESP32-C3 和 DHT11，确认 DHT11 数据线接在 GPIO4。
2. 在宿主机读取 USB 串口样本，生成 `data/raw/esp32_usb_samples.csv`。
3. 获取 Open-Meteo 当前数据与 7 天历史数据，生成 `data/raw/weather_observations.csv` 和 `data/raw/current_weather_combined.json`。
4. 在 `ubuntu22` distrobox 内启动 HDFS、YARN、HBase 和 HBase Thrift。
5. 上传历史数据到 HDFS，使用 Hive 建外部表查询，使用 Spark 生成统计结果。
6. 将三站当前数据写入 HBase 表 `weather_current`。
7. 启动 FastAPI 服务并打开可视化页面。

常用命令如下：

```bash
uvx --with pyserial python backend/scripts/read_esp32_usb.py \
  $ESP32_PORT \
  --count 10 \
  --out data/raw/esp32_usb_samples.csv

python backend/scripts/fetch_open_meteo.py data/raw
python backend/scripts/merge_current_weather.py
./scripts/start-bigdata-services.sh
./scripts/upload-weather-to-hdfs.sh data/raw/weather_observations.csv
./scripts/distro-bigdata.sh "hive -f $PROJECT_ROOT/backend/sql/create_weather_table.sql"
./scripts/distro-bigdata.sh "spark-submit --master local[2] $PROJECT_ROOT/backend/scripts/spark_weather_analysis.py /weather_lab/weathertextdb /weather_analysis $PROJECT_ROOT/data/processed"
```

## 六、实验结果与后续工作

本次完整版已完成从真实数据采集到可视化展示的闭环。唐山站当前温湿度由 ESP32-C3 与 DHT11 采集，并通过 USB 串口进入宿主机；唐山、北京、上海一周历史气象数据来自 Open-Meteo Archive API；历史数据写入 HDFS 后可由 Hive 外部表查询，并由 Spark 生成站点汇总、日汇总和小时序列；当前数据写入 HBase 表 `weather_current`；FastAPI 提供站点、指标、趋势、系统状态和 CSV 导出接口；前端提供总览、详细数据和系统状态三个视图。

当前无线热点与 ESP32 TCP 传输方案尚未作为主链路启用。原因是宿主机只有一块无线网卡，直接开启热点会断开当前上游 Wi-Fi；虚拟 AP 方案受当前上游 Wi-Fi 频段和 ESP32-C3 2.4GHz 支持限制影响。本系统暂以 USB 串口完成硬件数据链路，后续可在具备独立 2.4GHz 热点或第二块无线网卡时补充 TCP 传输。
