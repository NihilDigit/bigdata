# 气象大数据采集、分析与可视化系统课程设计报告

## 摘要

本课程设计实现了一个面向唐山、北京、上海三站的气象大数据采集、存储、分析与可视化系统。系统以 ESP32 与 DHT11 组成唐山自动气象站，采集真实温湿度数据；北京、上海站使用 Open-Meteo 当前气象数据作为基准，并加入小幅扰动形成连续实时数据。三站统一通过 WebSocket 向数据采集服务器推送观测记录，采集服务器将短期实时数据写入 HBase，将批量数据写入 HDFS。历史数据通过 Hive 外部表建仓，并由 Spark on YARN 计算站点统计、日统计、小时序列和 10 秒窗口均值。Web 层在保留课程指导文档数据链路的基础上，将 MTAS/ECharts 替换为 FastAPI、Next.js 和 Recharts，实现总览、详情和对比分析三个页面。系统最终完成了从传感器采集、无线传输、分布式存储、批处理分析到浏览器可视化展示的完整流程，并通过 HBase、HDFS、Hive、Spark、API 与前端页面的联合验证。

**关键词**：ESP32；DHT11；HBase；HDFS；Hive；Spark；FastAPI；Next.js

## 1. 系统简介

《大数据计算与可视化》课程设计要求完成一个气象大数据采集、分析与可视化系统，覆盖温湿度传感器采集、Wi-Fi 数据传输、HDFS 与 HBase 存储、Hive 数据仓库、Spark on YARN 分析以及 Web 可视化展示。本项目按照该流程实现完整系统，并在可视化服务和实时站点接入方式上做了工程化改进。

系统固定展示三个气象站点：

| 站点 | 作用 | 实时数据来源 | 历史数据来源 |
|---|---|---|---|
| 唐山 | 硬件演示站 | ESP32 + DHT11 采集温湿度，Open-Meteo 补齐气压、风速、风向和天气码 | Open-Meteo Archive API |
| 北京 | 对比站 | Open-Meteo 当前值 + 小幅扰动 | Open-Meteo Archive API |
| 上海 | 对比站 | Open-Meteo 当前值 + 小幅扰动 | Open-Meteo Archive API |

指导文档中的基础架构为“自动气象站 -> 数据采集服务器 -> HBase/HDFS -> Hive/Spark -> MTAS/ECharts”。本系统保留这条大数据链路，但将具体实现调整为：

```text
ESP32 + DHT11 / Open-Meteo 模拟站
        |
        | WebSocket JSON
        v
Python 数据采集服务器
        |
        |-----------------------> HBase realtime_weather  保存短期实时数据
        |
        |-----------------------> HDFS /weathertextdb     保存批量观测数据
                                      |
                                      v
                                Hive weather_table
                                      |
                                      v
                                Spark on YARN
                                      |
                                      v
                        /weather_analysis 与 data/processed/*.json
                                      |
                                      v
                            FastAPI + Next.js 可视化
```

本项目的主要改进包括：

- 将 ESP32 由 TCP Server 改为 WebSocket Client，主动连接采集服务器，减少热点 IP 变化带来的接入问题。
- 北京、上海站使用与唐山站相同的 WebSocket 协议接入，三站数据模型统一。
- 气压、风速、风向和天气码使用 Open-Meteo 真实气象数据，不使用纯随机数。
- FastAPI 替代 MTAS，提供结构化 REST API。
- Next.js + Recharts 替代静态 HTML + ECharts，页面拆分为总览、详情、对比分析三页。
- 提供一键启动脚本，统一启动 HDFS、YARN、HBase、采集服务、模拟站、Spark 分析服务、FastAPI 和 Next.js。

## 2. 环境搭建

### 2.1 硬件环境

唐山自动气象站由 ESP32 开发板和 DHT11 温湿度传感器组成。ESP32 通过 USB 连接电脑，用于供电、刷写程序和串口调试；DHT11 通过 VCC、GND 和 DATA 三线连接 ESP32。

![ESP32 与 DHT11 实物连接](../esp32.jpg)

图 2-1 ESP32 与 DHT11 温湿度传感器实物连接

实物连接关系如下：

| DHT11 模块 | ESP32 |
|---|---|
| VCC | 3.3V |
| GND | GND |
| DATA | GPIO18 |

板载程序检查结果显示，ESP32 当前 `main.py` 中的 `DHT_PIN = 18`，与实物连接一致。DHT11 模块指示灯点亮，说明供电正常。ESP32 运行 MicroPython 程序，每秒读取一次温湿度，并将采样结果通过 Wi-Fi 推送至采集服务器。

需要说明的是，仓库中的 ESP32 模板程序保留了默认 `DHT_PIN = 4`，上传脚本会根据环境变量生成实际写入板子的 `main.py`。本次实机检查确认板载程序已改为 GPIO18，运行时日志显示唐山站连接到采集服务器 `10.167.143.18:8080`，ESP32 自身获得的局域网地址为 `10.167.143.124`。因此报告中的硬件接线、板载程序和串口日志三者是一致的。

### 2.3 一键启动流程

系统提供 `scripts/start-all.sh` 作为一键启动入口：

```bash
./scripts/start-all.sh
./scripts/start-all.sh status
./scripts/start-all.sh stop
./scripts/start-all.sh restart
```

脚本默认端口如下：

| 服务 | 默认端口 |
|---|---:|
| FastAPI | 8008 |
| Next.js | 3000 |
| 采集 WebSocket Server | 8080 |
| Spark 分析常驻服务 | 18081 |
| HBase Thrift | 9090 |

启动顺序为：HDFS、YARN、HBase、HBase Thrift、采集服务器、北京模拟站、上海模拟站、Spark 分析服务、FastAPI 与 Next.js。脚本会等待 HDFS safe mode 关闭，并等待 HBase Thrift 可访问后再启动采集服务。若需要启动后自动触发一次 Spark 重算，可执行：

```bash

```

```bash
./scripts/upload-esp32-weatherstation.sh
```

## 3. 自动气象站与实时数据采集

### 3.1 ESP32 采集程序

ESP32 端程序位于 `esp32/weatherstation_main.py`。程序启动后完成三个步骤：

1. 连接指定 Wi-Fi 热点。
2. 使用 `dht.DHT11(Pin(DHT_PIN, pull=Pin.PULL_UP))` 读取温湿度。
3. 通过 WebSocket 连接采集服务器，并每秒发送一次 JSON 数据。

MicroPython 端没有额外引入 WebSocket 客户端库，而是使用 `socket` 手工完成 WebSocket 握手和文本帧发送。客户端先发送 `Upgrade: websocket` 请求，收到 `101 Switching Protocols` 后进入采样循环。发送文本帧时按照 WebSocket 协议对客户端 payload 加 mask，mask 由 `sample_seq` 和 `time.ticks_ms()` 生成。这样可以在 MicroPython 环境中保持依赖简单，同时满足服务器端 `websockets` 库对标准客户端帧的要求。

唐山站发送的数据格式如下：

```json
{
  "station_id": "tangshan",
  "station_name": "唐山",
  "temperature": 24,
  "humidity": 36,
  "sample_seq": 137
}
```

其中 `sample_seq` 为 ESP32 端递增样本号，用于证明数据来自连续实时采样。DHT11 为低成本温湿度传感器，适合课程设计中的真实数据采集演示；其温湿度读数精度有限，因此报告和页面只将其作为实时采集链路证据，不把它解释为高精度气象站。

串口调试输出分为两类：`serial_json` 表示本地已经读到 DHT11 数据并完成 JSON 组装，`send` 表示该 JSON 已经通过 WebSocket 发出。截图中 `sample_seq` 连续增长，说明采样循环没有停顿；详情页湿度曲线出现两个峰值，则说明串口、WebSocket、HBase 和前端实时查询之间的数据链路已贯通。

### 3.2 三站统一 WebSocket 接入

指导文档中 ESP32 作为 TCP Server，服务器端作为 TCP Client 连接 ESP32。该方式要求服务器知道 ESP32 的实时 IP，演示现场热点变化时容易失效。本系统将接入方式改为 WebSocket Client 主动推送：

- ESP32 作为唐山站 WebSocket Client。
- 北京、上海由 `backend/scripts/openmeteo_station_client.py` 作为 WebSocket Client。
- 采集服务器 `backend/scripts/weather_dataserver.py` 作为 WebSocket Server 监听 `0.0.0.0:8080`。

这种方式使三站接入协议统一，也便于采集服务器按同一套逻辑解析、补齐、入队和写入。

北京、上海站每秒推送一次数据。它们先从 Open-Meteo 当前天气接口获取真实基准值，再加入小幅扰动，使曲线在短时间内连续变化。扰动范围如下：

| 字段 | 扰动范围 |
|---|---:|
| 温度 | ±0.3 摄氏度 |
| 湿度 | ±2% |
| 气压 | ±0.5 hPa |
| 风速 | ±0.5 m/s |
| 风向 | ±5 度 |

天气码保留 Open-Meteo 当前天气码，不做随机替换。

### 3.3 统一数据模型

采集服务器最终形成统一九列气象记录：

```csv
collect_time,station_id,station_name,temperature,humidity,pressure,wind_speed,wind_direction,weather_code
```

字段含义如下：

| 字段 | 含义 | 单位 |
|---|---|---|
| `collect_time` | 采集时间，使用 Asia/Shanghai 时区 | - |
| `station_id` | 站点 ID | `tangshan` / `beijing` / `shanghai` |
| `station_name` | 站点中文名 | 唐山 / 北京 / 上海 |
| `temperature` | 温度 | 摄氏度 |
| `humidity` | 相对湿度 | % |
| `pressure` | 地面气压 | hPa |
| `wind_speed` | 10 米风速 | m/s |
| `wind_direction` | 10 米风向 | 度 |
| `weather_code` | 天气代码 | Open-Meteo WMO weather code |

唐山站硬件只提供温度和湿度，采集服务器会使用 Open-Meteo 当前值补齐气压、风速、风向和天气码。这样既保留真实硬件采集，又保证三站记录结构一致，便于 HDFS、Hive、Spark 和前端统一处理。

## 4. 数据采集服务器与存储模块

### 4.1 采集服务器设计

采集服务器位于 `backend/scripts/weather_dataserver.py`，负责接收三站 WebSocket 消息、补齐字段、记录本地 CSV，并并行写入 HBase 与 HDFS。

服务器内部把实时写入和批量归档分开处理：

- WebSocket handler 只负责解析消息、补齐字段、打印日志和入队。
- HBase 写入队列用于实时数据，收到记录后尽快写入 `realtime_weather`。
- HDFS 写入队列按批次写入 `/weathertextdb/live_*.csv`。
- Open-Meteo 当前值按固定周期刷新，避免每条数据都请求外部 API。

这种设计避免了 HBase/HDFS 写入阻塞 WebSocket 接收，使三站每秒推送时仍能保持接收稳定。

采集服务器支持两种输入格式：一种是 ESP32 当前使用的 JSON 文本，另一种是兼容早期实验的 CSV 行。服务器收到消息后统一转换为 `WeatherRecord`，并在接收时刻生成 `collect_time`。如果消息中没有气压、风速、风向或天气码，服务器会从 Open-Meteo 当前值缓存中补齐，并加入小范围扰动，使唐山站也能进入后续九列统一模型。

### 4.2 HBase 实时存储

HBase 用于保存短期实时气象数据，满足前端当前读数和最近 150 秒趋势查询。表结构如下：

| 项目 | 设计 |
|---|---|
| 表名 | `realtime_weather` |
| 列族 | `data` |
| TTL | 150 秒 |
| 行键 | `<station_id>_<collect_time_compact>` |
| 写入方式 | HappyBase Thrift batch put |

列包括：

```text
data:collect_time
data:station_name
data:temperature
data:humidity
data:pressure
data:wind_speed
data:wind_direction
data:weather_code
data:sample_seq
```

HBase 与 HDFS 的职责不同：HBase 适合读取最近实时数据，HDFS 适合保存批量数据供 Hive 和 Spark 分析。采集服务器因此采用双写方式，不用 HDFS 承担实时查询，也不让 HBase 承担长期分析。

HBase row key 的格式为 `station_id + 紧凑时间戳`，例如 `tangshan_202606252214...`。这种行键使同一站点的实时数据天然按时间排列，FastAPI 查询详情页时可以用 `row_prefix=tangshan_` 扫描唐山站最近数据，再按最新时间向前截取 150 秒窗口。HBase 表的列族 TTL 设置为 150 秒，旧实时记录会自动过期，避免演示过程中实时表无限增长。

### 4.3 HDFS 批量存储

HDFS 保存历史数据和实时批次文件。历史数据主文件为：

```text
/weathertextdb/weather_observations.csv
```

实时采集服务继续向同一目录写入：

```text
/weathertextdb/live_*.csv
```

实时批次文件由采集服务器按 `flush-size` 聚合后写入 HDFS。默认启动脚本把 `FLUSH_SIZE` 设为 30，即三站每秒推送时大约每 10 秒产生一个 HDFS 实时批次。HDFS 批次文件不写表头，字段顺序与 Hive 外部表一致，便于 Spark 直接按固定 schema 读取。

本地历史原始数据位于 `data/raw/weather_observations.csv`。该文件包含表头，共 1008 条历史小时记录，范围为 `2026-06-11T00:00` 到 `2026-06-24T23:00`，三站各 336 条。实时采集本地缓存位于 `data/raw/live_weather_observations.csv`，用于开发调试和采集过程留痕。

历史 CSV 上传命令为：

```bash
./scripts/upload-weather-to-hdfs.sh data/raw/weather_observations.csv
```

## 5. Hive 数据仓库与 Spark 分析

### 5.1 Hive 外部表

Hive 用于在 HDFS CSV 文件之上建立气象数据仓库。建表脚本位于 `backend/sql/create_weather_table.sql`，表名为 `weather_table`，位置为 `/weathertextdb`。

表结构如下：

```sql
CREATE EXTERNAL TABLE weather_table (
  collect_time STRING,
  station_id STRING,
  station_name STRING,
  temperature DOUBLE,
  humidity DOUBLE,
  pressure DOUBLE,
  wind_speed DOUBLE,
  wind_direction DOUBLE,
  weather_code INT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LOCATION '/weathertextdb';
```

与指导文档中的七列格式相比，本系统增加了 `station_id` 和 `weather_code`。`station_id` 便于接口和前端稳定筛选站点，`weather_code` 用于前端天气标签和图标映射。

### 5.2 Spark on YARN 分析

Spark 分析脚本位于 `backend/scripts/spark_weather_analysis.py`，运行方式如下：

```bash
SOURCE=hive ./scripts/run-spark-analysis-yarn.sh
```

脚本读取 Hive 表或 HDFS CSV 后，完成以下分析：

| 输出 | 说明 |
|---|---|
| `station_summary.json` | 三站统计摘要 |
| `daily_summary.json` | 三站逐日统计 |
| `hourly_series.json` | 历史小时序列 |
| `window_mean_series.json` | 10 秒窗口均值序列 |

Spark 同时把结果写入 HDFS：

```text
/weather_analysis/summary
/weather_analysis/daily_summary
/weather_analysis/hourly_series
/weather_analysis/window_mean
```

统计摘要字段包括记录数、平均温度、最低温度、最高温度、平均湿度、平均气压、平均风速、气压变化量、最大风速和主导风向。其中主导风向不是简单算术平均，而是把风向角转换为正弦、余弦分量后做向量平均，避免 359 度与 1 度被错误平均为 180 度。

当前历史数据统计摘要如下：

| 站点 | 记录数 | 平均温度 | 最低温度 | 最高温度 | 平均湿度 | 平均气压 | 平均风速 | 气压变化量 | 最大风速 | 主导风向 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| 北京 | 336 | 25.08 | 16.70 | 33.30 | 62.20 | 1001.42 | 1.74 | 11.70 | 4.58 | 79.14 |
| 上海 | 336 | 24.66 | 19.60 | 34.50 | 80.98 | 1007.41 | 2.50 | 13.30 | 5.01 | 94.74 |
| 唐山 | 336 | 23.12 | 15.70 | 30.10 | 72.84 | 1003.41 | 2.23 | 10.00 | 5.61 | 154.66 |

### 5.3 Spark 刷新服务

为了方便演示，系统增加了 Spark 常驻分析服务。前端“对比分析”页面可以手动触发 Spark 重算，FastAPI 调用常驻服务提交刷新请求。刷新过程中旧 JSON 缓存继续对外提供数据，待 Spark 完成后再替换为最新结果。这样页面不会因为 Spark 作业运行而空白。

常驻服务监听 `127.0.0.1:18081`，提供 `/health`、`/status` 和 `/refresh` 三个接口。服务启动时创建 SparkSession，并保存 YARN application id；每次刷新请求进入后台线程执行，状态写入 `.runtime/spark-analysis-status.json`，日志写入 `.runtime/spark-analysis.log`。前端轮询 `/api/analysis/refresh/status`，根据 `progress_percent`、`progress_label`、`job_id`、`application_id` 和 `exit_code` 渲染进度条和日志尾部。

本次运行中的 Spark 输出显示，历史小时数据为 1008 条，实时与历史合计有效记录为 23235 条；刷新完成后生成 3 行站点摘要、42 行日统计、1008 行小时序列和 1425 行 10 秒窗口均值。该结果说明 Spark 分析既保留课程所需的历史批处理结果，也能把实时批次纳入短窗口均值计算。

## 6. Web 接口与可视化

### 6.1 FastAPI 接口

指导文档使用 MTAS 发布静态页面、实时数据和均值数据。本系统将该部分替换为 FastAPI。FastAPI 不负责静态页面构建，只提供结构化数据接口，前端由 Next.js 独立运行。

主要接口如下：

| 接口 | 用途 |
|---|---|
| `GET /api/stations/current` | 从 HBase 读取三站当前读数 |
| `GET /api/stations/{station_id}/live?seconds=150` | 从 HBase 读取站点最近实时序列 |
| `GET /api/analysis/trends` | 读取 Spark 历史趋势或窗口均值 |
| `GET /api/analysis/summary` | 读取 Spark 统计摘要 |
| `POST /api/analysis/refresh` | 触发 Spark 重算 |
| `GET /api/records` | 读取历史原始记录 |
| `GET /api/hdfs/records` | 读取 HDFS 实时批次记录 |
| `GET /api/export/weather_observations.csv` | 导出历史 CSV |

当前数据接口直接读取 HBase Thrift。如果 HBase 当前表缺少某个站点，接口返回服务不可用错误，避免静默回退到旧 JSON 导致页面展示假实时数据。

接口层的实时与历史数据来源不同：`/api/stations/current` 和 `/api/stations/{station_id}/live` 直接访问 HBase Thrift；`/api/analysis/*` 读取 Spark 生成的本地 JSON 缓存；`/api/hdfs/records` 通过 `hdfs dfs -cat` 读取最新 `live_*.csv` 批次；`/api/export/weather_observations.csv` 则导出本地历史 CSV。这样的拆分使页面能够同时展示实时状态、批处理结果和原始记录。

## 8. 问题定位与改进

### 8.1 ESP32 接入方式调整

指导文档中的 ESP32 作为 TCP Server 方案需要服务器主动连接 ESP32。实际调试中，热点分配的 IP 会变化，且主机防火墙可能阻止入站连接。本项目改为 ESP32 主动连接采集服务器的 WebSocket 方案。这样只需在 ESP32 程序中配置服务器 IP，采集服务器无需逐个连接气象站，三站协议也更一致。

### 8.2 HBase 写入方式优化

早期实现如果每批调用 `hbase shell`，并反复执行建表或扫描操作，会拖慢实时采集。最终实现改用 HappyBase Thrift 直接批量写入，采集服务只在启动时确保表存在，运行时按记录写入 HBase。

### 8.3 Spark 分析扩展

指导文档重点要求 10 秒窗口均值。本项目在此基础上增加了站点摘要、日统计、小时序列、气压变化量、最大风速和主导风向。主导风向使用向量平均，避免普通平均对环形角度不适用的问题。

### 8.4 Web 层替换

FastAPI 和 Next.js 的替换使接口更清晰，页面更容易拆分，也便于演示时手动触发 Spark 重算。这个改动属于课程指导文档允许的改进项，数据来源仍然来自 HBase、HDFS、Hive 和 Spark。

实时记录表调试时出现过 React “key 不唯一”警告。原因是 HDFS 批次记录中可能出现同一站点同一采集时间的多条记录，单独使用 `station_id + collect_time` 作为表格行 key 不够稳定。最终改为 `station_id + collect_time + sample_seq + index` 的组合 key，使实时表格在频繁刷新时能保持组件身份稳定，避免行被重复或遗漏。

## 9. 结论

本课程设计完成了气象数据采集、传输、存储、处理和可视化的完整流程。唐山站使用 ESP32 与 DHT11 采集真实温湿度数据，北京、上海使用 Open-Meteo 当前真实数据加扰动形成对比站。三站统一推送到 WebSocket 采集服务器，服务器并行写入 HBase 和 HDFS。HBase 负责短期实时查询，HDFS 负责批量存储；Hive 在 HDFS 上建立外部表，Spark on YARN 完成多项统计分析；FastAPI 和 Next.js 将实时数据与分析结果展示在浏览器中。

系统最终实现了课程要求中的关键组件：ESP32-C3、DHT11、Wi-Fi 传输、HDFS、HBase、Hive、Spark on YARN 和 Web 可视化。同时，系统在实时接入、Open-Meteo 数据来源、Spark 分析内容和前端展示方式上做了改进。剩余限制主要包括 DHT11 精度有限、演示现场热点 IP 可能变化、北京和上海站仍属于软件模拟站，以及 Open-Meteo 数据获取依赖网络。后续可以加入更多实物气象站、使用更高精度传感器，并将 Spark 分析调度进一步自动化。
