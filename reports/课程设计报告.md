# 气象大数据采集、分析与可视化系统课程设计报告

## 摘要

本课程设计实现面向唐山、北京、上海三站的气象大数据采集、存储、分析与可视化系统。唐山站由 ESP32-WROVER-E 与 DHT11 组成自动气象站，采集真实温湿度；北京、上海站以 Open-Meteo 当前气象数据叠加小幅扰动形成连续实时数据。三站经 WebSocket 推送至采集服务器，实时数据写入 HBase，批量数据写入 HDFS。Hive 在 HDFS 之上建立外部表，Spark on YARN 完成站点统计、日统计、小时序列与 10 秒窗口均值计算。可视化层以 FastAPI、Next.js 与 Recharts 替换 MTAS 与 ECharts，提供总览、详情与对比分析三个页面。系统打通从传感器采集到浏览器可视化的完整流程，并通过 HBase、HDFS、Hive、Spark 与前端页面的联合验证。

## 1. 系统简介

本系统是一个面向唐山、北京、上海三站的气象大数据采集、存储、分析与可视化系统，覆盖温湿度传感器采集、Wi-Fi 数据传输、HDFS 与 HBase 存储、Hive 数据仓库、Spark on YARN 分析与 Web 可视化展示。唐山站为硬件演示站，由 ESP32-WROVER-E 与 DHT11 组成自动气象站；北京、上海为对比站，以 Open-Meteo 数据形成连续实时输入。

## 2. 环境搭建

### 2.1 硬件环境

唐山自动气象站由 ESP32-WROVER-E 开发板和 DHT11 温湿度传感器组成，两者插在面包板上通过杜邦线连接。ESP32-WROVER-E 搭载乐鑫 ESP32 芯片，集成 2.4 GHz Wi-Fi 与双核处理器，支持 MicroPython 固件。开发板通过 USB 连接电脑，用于供电、刷写程序和串口调试；DHT11 通过 VCC、GND 和 DATA 三线连接 ESP32。

## 3. 数据采集与传输

### 3.1 自动气象站采集

唐山站负责真实温湿度采集。自动气象站以 ESP32-WROVER-E 为核心，通过 DHT11 每秒读取一次温湿度，经 Wi-Fi 推送至采集服务器。采集程序完成三步：连接指定热点，初始化 DHT11 并读取温湿度，建立 WebSocket 连接后每秒发送一条 JSON 记录。

MicroPython 没有现成的 WebSocket 客户端库，采集程序用 `socket` 手工完成 WebSocket 握手与文本帧发送。握手阶段发送 HTTP `Upgrade: websocket` 请求，待服务器返回 `101 Switching Protocols` 后进入 WebSocket 通信；发送文本帧时按协议对客户端 payload 加 4 字节 mask。握手与帧发送的核心代码如下：

![ESP32 WebSocket 握手与文本帧发送](assets/code_esp32_ws.png)

图 3-1 ESP32 WebSocket 握手与文本帧发送

选择手写 WebSocket 而非引入客户端库，是为了在 MicroPython 环境中保持依赖简单、固件体积小；mask 使用样本号与 `time.ticks_ms()` 生成，既能满足协议对客户端帧的掩码要求，又顺带携带采样序号信息，避免额外字段。服务器端 `websockets` 库按标准客户端帧解析，双方握手与传输均正常。

每条记录携带站点 ID、站点名、温度、湿度与递增样本号 `sample_seq`：

```json
{
  "station_id": "tangshan",
  "station_name": "唐山",
  "temperature": 24,
  "humidity": 36,
  "sample_seq": 137
}
```

`sample_seq` 为 ESP32 端递增样本号，用于验证数据来自连续实时采样。DHT11 为低成本温湿度传感器，用于真实温湿度采集演示；其读数精度有限，仅作为实时采集链路证据，不作为高精度气象观测。

### 3.2 三站统一 WebSocket 接入

本系统采用 WebSocket Client 主动推送的接入方式：各站点作为 WebSocket Client 主动连接采集服务器，采集服务器作为 WebSocket Server 监听 `0.0.0.0:8080` 统一接收。相比由服务器反向连接站点的方式，主动推送不依赖站点的实时 IP，热点分配地址变化时仍可接入。三站接入协议统一，采集服务器按同一套逻辑解析、补齐、入队与写入。

唐山站由 ESP32 担任 WebSocket Client；北京、上海站由 Open-Meteo 模拟客户端担任。Open-Meteo 是免费的开放气象数据 API，提供全球站点的历史与实时气象预报数据，无需密钥即可调用。本系统使用其 Forecast API 获取当前天气基准值，使用 Archive API 获取历史小时数据。北京、上海站每秒推送一次数据，先从 Open-Meteo 当前天气接口获取真实基准值，再叠加小幅扰动，使曲线在短时间内连续变化。扰动范围如下：

| 字段 | 扰动范围 |
|---|---:|
| 温度 | ±0.3 摄氏度 |
| 湿度 | ±2% |
| 气压 | ±0.5 hPa |
| 风速 | ±0.5 m/s |
| 风向 | ±5 度 |

天气码保留 Open-Meteo 当前天气码，不做随机替换。

### 3.3 统一数据模型

采集服务器将三站数据归并为统一九列气象记录：

```csv
collect_time,station_id,station_name,temperature,humidity,pressure,wind_speed,wind_direction,weather_code
```

字段含义如下：

| 字段 | 含义 | 单位 |
|---|---|---|
| `collect_time` | 采集时间，使用 Asia/Shanghai 时区 | - |
| `station_id` | 站点 ID | `tangshan` / `beijing` / `shanghai` |
| `station_name` | 站点中文名 | 唐山 / 北京 / 上海 |
| `temperature` | 温度 | 摄氏度 |
| `humidity` | 相对湿度 | % |
| `pressure` | 地面气压 | hPa |
| `wind_speed` | 10 米风速 | m/s |
| `wind_direction` | 10 米风向 | 度 |
| `weather_code` | 天气代码 | Open-Meteo WMO weather code |

唐山站硬件只提供温度和湿度，采集服务器使用 Open-Meteo 当前值补齐气压、风速、风向和天气码。这样既保留真实硬件采集，又保证三站记录结构一致，便于 HDFS、Hive、Spark 和前端统一处理。

## 4. 数据存储

采集服务器收到三站推送后，将实时数据写入 HBase、批量数据写入 HDFS，二者按职责分工：HBase 存放短期实时数据供前端当前读数与近 150 秒趋势查询，HDFS 存放批量数据供 Hive 与 Spark 分析。

### 4.1 采集服务器

采集服务器作为 WebSocket Server 接收三站消息，完成字段补齐与并行写入。为避免存储写入阻塞 WebSocket 接收，服务器将实时写入与批量归档分离，内部结构如下图所示：

![采集服务器内部结构](assets/diag_ingest.png)

图 4-1 采集服务器内部结构

WebSocket handler 只负责解析消息、补齐字段与入队，不直接触碰 HBase 或 HDFS。实时队列收到记录后尽快写入 HBase；批量队列按 `flush-size` 聚合后写入 HDFS；Open-Meteo 当前值按固定周期刷新缓存，避免每条数据都请求外部 API。

实时与批量分离的关键在于 HBase 与 HDFS 的写入速率差异。HBase 通过 Thrift 批量 put，能承载三站每秒共 3 条的写入节奏；HDFS 每次写入都要经过 NameNode 协调与 `hdfs dfs -put` 落盘，单条写入延迟远高于 HBase，若按 HBase 的频率同步写 HDFS，写入速度跟不上到达速度，数据会在内存中积压。因此实时数据每条立即入 HBase 队列，HDFS 侧则攒满 `flush-size`（默认 30 条，约 10 秒一个批次）再一次性 put，使 HDFS 的写入节奏与其吞吐能力匹配。

服务器收到 JSON 消息后统一转换为内部记录，并在接收时刻生成 `collect_time`。若消息缺少气压、风速、风向或天气码，从 Open-Meteo 当前值缓存补齐并加小范围扰动，使唐山站也进入九列统一模型。

HBase 用于短期实时数据存储，列族 TTL 设为 150 秒，旧记录自动过期，避免实时表无限增长。建表与批量写入的核心代码如下：

表结构如下：

| 项目 | 设计 |
|---|---|
| 表名 | `realtime_weather` |
| 列族 | `data` |
| TTL | 150 秒 |
| 行键 | `<station_id>_<collect_time_compact>` |

行键格式为 `station_id + 紧凑时间戳`，例如 `tangshan_202606252214...`，使同一站点的实时数据天然按时间排列。前端查询详情页时以 `row_prefix=tangshan_` 扫描唐山站最近数据，再按最新时间向前截取 150 秒窗口，无需读取后排序。

采集服务器采用双写方式：不用 HDFS 承担实时查询，也不让 HBase 承担长期分析。

## 5. 数据分析与处理

历史与实时数据落入 HDFS 后，由 Hive 建立数据仓库，由 Spark on YARN 完成统计分析，结果供前端可视化消费。

### 5.1 Hive 外部表

Hive 在 HDFS 的 CSV 文件之上建立外部表 `weather_table`，位置为 `/weathertextdb`，使历史主文件与实时批次文件统一纳入一张表。表结构如下：

```sql
CREATE EXTERNAL TABLE weather_table (
  collect_time STRING,
  station_id STRING,
  station_name STRING,
  temperature DOUBLE,
  humidity DOUBLE,
  pressure DOUBLE,
  wind_speed DOUBLE,
  wind_direction DOUBLE,
  weather_code INT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LOCATION '/weathertextdb';
```

相比基础七列格式，本表增加 `station_id` 与 `weather_code`：`station_id` 供接口与前端稳定筛选站点，`weather_code` 供前端天气标签与图标映射。

### 5.3 Spark 常驻服务

系统设置 Spark 常驻分析服务，托管 SparkSession 并复用同一个 YARN application，避免每次重算都重新提交作业。前端对比分析页的“手动刷新”按钮经 FastAPI 调用常驻服务触发重算；刷新期间旧 JSON 缓存继续对外提供数据，Spark 完成后再替换为最新结果，页面不会因作业运行而空白。

常驻服务提供 `/health`、`/status` 与 `/refresh` 三个接口。每次刷新在后台线程执行，状态与日志供前端轮询，前端据 `progress_percent`、`application_id`、`exit_code` 等字段渲染进度条与日志尾部。

本次运行中，历史小时数据 1008 条，实时与历史合计有效记录 23235 条；刷新完成后生成 3 行站点摘要、42 行日统计、1008 行小时序列与 1425 行 10 秒窗口均值。Spark 分析既覆盖历史批处理，也把实时批次纳入短窗口均值计算。

## 6. 数据可视化

可视化层将 HBase 的实时数据与 Spark 的分析结果呈现到浏览器。系统以 FastAPI 提供结构化 REST API，以 Next.js + Recharts 实现前端页面，二者独立运行：FastAPI 只负责数据接口，前端单独构建与部署。

### 6.1 FastAPI 接口

## 8. 改进与加分项说明

本系统在基础要求之上做了以下改进。

### 8.1 自动气象站核心板升级

自动气象站核心板由 ESP32-C3 改为 ESP32-WROVER-E，在面包板上搭接，获得更高的处理性能与更充裕的内存，保障 MicroPython 采样与 WebSocket 发送的稳定性。

### 8.2 ESP32 接入方式调整

ESP32 由 TCP Server 改为 WebSocket Client，主动连接采集服务器。相比由服务器反向连接站点的方式，主动推送不依赖站点的实时 IP，热点分配地址变化时仍可接入；三站接入协议统一，采集服务器按同一套逻辑解析与写入。

### 8.3 HBase 写入方式优化

采集服务改用 HappyBase Thrift 直接批量写入，仅在启动时确保表存在，运行时按记录写入 HBase，避免逐批调用 `hbase shell` 与反复建表、扫描拖慢实时采集。

### 8.4 Spark 分析扩展与自动执行

Spark 分析在 10 秒窗口均值基础上增加站点摘要、日统计、小时序列、气压变化量、最大风速与主导风向；主导风向采用向量平均，避免普通平均对环形角度不适用的问题。Spark 分析由常驻服务托管，前端可手动触发重算并展示进度；服务复用同一个 YARN application，避免反复提交作业带来的资源开销，实现数据处理自动执行与防止反复执行。

### 8.5 Web 层替换

以 FastAPI 替代 MTAS，以 Next.js + Recharts 替代 HTML/CSS/JS + ECharts，接口结构化、页面拆分为总览/详情/对比分析三页，并支持前端触发 Spark 重算。数据来源仍为 HBase、HDFS、Hive 与 Spark，数据链路不变。

### 8.6 实时记录表 key 稳定性

HDFS 批次记录中可能出现同一站点同一采集时间的多条记录，单独使用 `station_id + collect_time` 作为表格行 key 不够稳定。实时记录表改用 `station_id + collect_time + sample_seq + index` 的组合 key，在频繁刷新时保持组件身份稳定，避免行被重复或遗漏。

## 9. 结论

本系统完成了气象数据采集、传输、存储、处理与可视化的完整流程。唐山站由 ESP32-WROVER-E 与 DHT11 采集真实温湿度，北京、上海以 Open-Meteo 当前数据加扰动形成对比站；三站经 WebSocket 推送至采集服务器，并行写入 HBase 与 HDFS。HBase 承担短期实时查询，HDFS 承担批量存储；Hive 在 HDFS 上建立外部表，Spark on YARN 完成站点统计、日统计、小时序列与 10 秒窗口均值；FastAPI 与 Next.js 将实时数据与分析结果呈现于浏览器。

系统覆盖了 ESP32-WROVER-E、DHT11、Wi-Fi 传输、HDFS、HBase、Hive、Spark on YARN 与 Web 可视化等关键组件，并在实时接入、Open-Meteo 数据来源、Spark 分析内容与前端展示方式上做了改进。剩余限制包括 DHT11 精度有限、热点分配 IP 可能变化、北京与上海站仍属软件模拟站、Open-Meteo 数据获取依赖网络。后续可加入更多实物气象站、采用更高精度传感器，并将 Spark 分析调度进一步自动化。
