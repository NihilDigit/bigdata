# Weather Lab 气象大数据计算与可视化系统设计文档

打开主页时，屏幕上只有三类信息：一张合规中国轮廓图，三个站点的天气点位，一个当前站点的五指标读数框。用户点击北京、唐山、上海任一站点，地图下方的温度、湿度、气压、风速、风向立即切换。需要查看趋势、统计和原始记录时，再进入“详细数据”页面。

这个界面背后是一条端到端数据链路：唐山站的温湿度来自 ESP32-C3 和 DHT11，气压、风速、风向来自 Open-Meteo 当前天气接口；北京、上海的实时数据来自 Open-Meteo。开发截止日前的历史气象数据预先导入 HDFS，Hive 建外部表，Spark 生成趋势与统计结果，FastAPI 提供给 Next.js 前端。

## 1. 系统边界

课程指导文档要求完成气象数据采集、传输、存储、处理和可视化。Weather Lab 保留这条主线：

```text
ESP32-C3 + DHT11
-> Python 采集服务
-> HBase 实时数据
-> HDFS 历史数据
-> Hive 外部表
-> Spark 批处理分析
-> FastAPI
-> Next.js 可视化界面
```

Web 层做了三处调整，需要在课程报告中列为改进项。

第一，FastAPI 替换 MTAS。FastAPI 更适合前后端分离，接口结构也更容易描述和测试。

第二，Next.js、shadcn/ui、Recharts 替换原指导方案里的静态页面和 ECharts。这个选择让页面组件更清楚，视觉也更接近现代数据产品。课程报告中应说明这是可视化界面的实现改进。

第三，气压、风速、风向不再由随机数生成，改用 Open-Meteo 的公开气象数据。DHT11 只能采集温湿度，其他气象指标来自公共数据源更合理，也更容易解释。

系统固定三个站点：

| 站点 | 页面角色 | 数据来源 |
|---|---|---|
| 唐山 | 硬件演示主站 | ESP32 温湿度 + Open-Meteo 气压、风速、风向 |
| 北京 | 对比站点 | Open-Meteo 当前数据和历史数据 |
| 上海 | 对比站点 | Open-Meteo 当前数据和历史数据 |

## 2. 数据定义

系统内部用一条观测记录描述某站点某一时刻的天气。

```csv
collect_time,station_id,station_name,temperature,humidity,pressure,wind_speed,wind_direction,weather_code
```

字段含义如下：

| 字段 | 含义 | 单位 |
|---|---|---|
| collect_time | 采集或统一后的时间 | Asia/Shanghai |
| station_id | 站点 ID | tangshan / beijing / shanghai |
| station_name | 站点中文名 | 唐山 / 北京 / 上海 |
| temperature | 2 米气温或 DHT11 温度 | 摄氏度 |
| humidity | 相对湿度 | % |
| pressure | 地面气压 | hPa |
| wind_speed | 10 米风速 | m/s |
| wind_direction | 10 米风向 | 度 |
| weather_code | 天气代码 | Open-Meteo weather code |

课程指导文档中的 Hive 表只有七个核心字段。为兼容课程要求，HDFS 中保留一份七列 CSV：

```csv
collect_time,station_name,temperature,humidity,pressure,wind_speed,wind_direction
```

扩展字段留给 API 和前端使用。这样 Hive、Spark 部分能按指导文档验收，前端又能在地图上展示天气状态。

Open-Meteo 当前接口需要这些字段：

```text
temperature_2m
relative_humidity_2m
surface_pressure
wind_speed_10m
wind_direction_10m
weather_code
```

天气代码在前端映射成短标签。地图点位只显示天气状态和温度，不显示全部指标。

```text
0: 晴
1, 2, 3: 少云 / 多云
45, 48: 雾
51, 53, 55: 毛毛雨
61, 63, 65: 雨
71, 73, 75, 77: 雪
80, 81, 82: 阵雨
95, 96, 99: 雷暴
```

## 3. 数据链路

### 3.1 实时链路

ESP32-C3 上电后连接 Wi-Fi，读取 DHT11 的温度、湿度，并用 TCP 发给采集服务。采集服务在服务器侧缓存 Open-Meteo 当前天气数据，再合并成完整观测记录。

唐山站实时记录由两部分组成：

```text
temperature, humidity
  来自 ESP32-C3 + DHT11

pressure, wind_speed, wind_direction, weather_code
  来自 Open-Meteo Current Weather API
```

北京、上海站不接硬件，直接读取 Open-Meteo 当前数据。

合并后的记录写入 HBase，供主页和详细数据页读取。记录也按批次写入 HDFS，后续进入 Hive 和 Spark。

### 3.2 历史链路

开发截止日前，先拉取三个站点的 Open-Meteo 历史数据，规范化为课程要求的 CSV 格式，然后上传到 HDFS。

```text
Open-Meteo Historical Weather API
-> CSV 规范化
-> HDFS /weathertextdb
-> Hive weather_table
-> Spark 分析任务
-> HDFS 分析结果
-> FastAPI
-> Next.js 详细数据页
```

这一步保证演示当天即使网络不稳定，详细数据页仍有历史趋势和统计结果。

### 3.3 实时追加

系统运行后，采集服务继续把合并后的实时记录追加到 HDFS。Spark 可以定时重算最近一段时间的聚合结果。

Open-Meteo 当前接口通常按分钟级更新。采集服务不应每秒请求该接口，建议每 5 到 15 分钟刷新一次，并在两次刷新之间复用最近值。DHT11 的温湿度仍可按 1 秒间隔采样。

## 4. 存储设计

### 4.1 HBase

HBase 保存短期实时记录，服务主页和详细数据页的“当前数据”“最近记录”。

表名：

```text
realtime_weather
```

列族：

```text
data
```

行键：

```text
station_id_timestamp
```

示例：

```text
tangshan_20260618104521
```

列包括：

```text
data:station_name
data:temperature
data:humidity
data:pressure
data:wind_speed
data:wind_direction
data:weather_code
data:collect_time
```

指导文档示例使用 150 秒 TTL。演示时可以改成 600 秒，避免页面刷新后没有足够点位。报告中说明 HBase 只保存短期实时数据即可。

### 4.2 HDFS

HDFS 保存历史和实时追加数据。

核心路径：

```text
/weathertextdb
```

该路径保存课程兼容 CSV，无表头：

```csv
2026-06-18 10:45:21,唐山,28.5,64,1003.6,2.39,200
```

Spark 输出建议放在独立路径：

```text
/weather_analysis/window_mean
/weather_analysis/summary
```

如果按指导文档命名，也可以使用：

```text
/weather_10secmean
```

## 5. Hive 表

Hive 给 HDFS 中的 CSV 加上表结构。最低实现只需要一张外部表。

```sql
CREATE EXTERNAL TABLE IF NOT EXISTS weather_table (
    collect_time TIMESTAMP COMMENT '采集时间',
    station_name STRING COMMENT '气象站名称',
    temperature DOUBLE COMMENT '温度，摄氏度',
    humidity DOUBLE COMMENT '相对湿度，百分比',
    pressure DOUBLE COMMENT '地面气压，hPa',
    wind_speed DOUBLE COMMENT '10 米风速，m/s',
    wind_direction DOUBLE COMMENT '10 米风向，角度'
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LOCATION '/weathertextdb';
```

指导文档里的验证 SQL 有拼写错误，实际应查询：

```sql
SELECT * FROM weather_table LIMIT 10;
```

## 6. Spark 分析

Spark 不直接面向前端用户。它负责把原始气象记录变成更适合观察的趋势和摘要。

### 6.1 窗口均值

窗口均值是指导文档明确要求的分析任务。秒级数据直接画图会有噪声，数量也偏多。Spark 将数据按固定时间窗口分组，再计算每组平均值。

例如唐山站每秒产生一条温度记录：

```text
10:00:01 28.1
10:00:02 28.2
10:00:03 28.0
...
10:00:09 28.4
```

按 10 秒窗口聚合后，输出一条记录：

```text
10:00:00 28.18
```

详细数据页里的趋势图可以切换粒度：

```text
10 秒
1 小时
1 天
```

课程验收优先完成 10 秒窗口均值。

### 6.2 统计摘要

详细数据页还需要一组容易理解的摘要指标：

```text
平均温度
最高温度
最低温度
平均湿度
平均气压
气压变化量
平均风速
最大风速
主导风向
```

这些指标适合放在“统计摘要”区，比单独展示 Spark 作业名更符合用户直觉。

### 6.3 多站点对比

多站点对比回答一个问题：同一时间范围内，北京、唐山、上海的同一指标有何差异。

前端允许选择指标：

```text
温度
湿度
气压
风速
风向
```

图表展示三条线。数据可以直接复用窗口均值结果。

### 6.4 风向计算

风向是角度，直接平均会出错。例如 350 度和 10 度的平均值不应是 180 度。

更合适的做法是向量平均：

```text
u = mean(cos(theta))
v = mean(sin(theta))
direction = atan2(v, u)
```

最低实现可以先用普通平均。报告中应说明风向字段的局限；时间允许时再改成向量平均。

### 6.5 防重复写入

Spark 定时运行时，不能一直 append 最近半小时结果，否则同一时间窗口会重复出现。

推荐按时间分区覆盖：

```text
/weather_analysis/window_mean/date=2026-06-18/hour=10
```

重算同一小时结果时覆盖该分区。API 查询时按时间范围读取对应分区。

## 7. FastAPI 接口

FastAPI 提供业务数据，不向前端暴露 HBase、HDFS、Spark 的运行状态。

主页需要一个聚合接口：

```text
GET /api/stations/current
```

返回三个站点当前天气。地图点位和下方五指标框都读这个接口。

```json
[
  {
    "id": "tangshan",
    "name": "唐山",
    "latitude": 39.63,
    "longitude": 118.18,
    "time": "2026-06-18T10:45:21+08:00",
    "temperature": 28.5,
    "humidity": 64,
    "pressure": 1003.6,
    "wind_speed": 2.39,
    "wind_direction": 200,
    "weather_code": 3
  }
]
```

详细数据页需要三个接口：

```text
GET /api/stations/{station_id}/live?seconds=150
GET /api/analysis/trends
GET /api/records
```

`/api/analysis/trends` 支持这些查询参数：

```text
station_id=tangshan | beijing | shanghai | all
metric=temperature | humidity | pressure | wind_speed | wind_direction
range=24h | 7d | custom
granularity=10s | 1h | 1d
```

## 8. 前端设计

前端遵循一个原则：界面按用户要看的天气信息组织，不按后端组件组织。HBase、HDFS、Hive、Spark 是报告和答辩中的实现说明，不放在主页上。

### 8.1 主页

主页只有标题、地图框、当前指标框。

```text
Weather Lab / 气象数据工作台

|                                                                                |

+--------------------------------------------------------------------------------+
```

地图框承担两件事：


当前指标框展示完整五指标。用户需要趋势和记录时，点击“查看详细数据”。


图表安排按气象指标分组：

```text
温湿度趋势：温度和湿度放在一起。
气压趋势：气压单独展示。
风况：风速和风向放在一起。
多站对比：同一指标比较三个城市。
```

### 8.3 视觉风格

页面使用浅色工作台风格。背景用冷灰白，文字用石墨黑，边界用低对比细线。主色用于选中站点、按钮和活动图线。

指标颜色建议：

```text
温度：低饱和橙红
湿度：低饱和蓝
气压：紫灰
风速：绿色
风向：靛蓝
```

避免这些视觉处理：

```text
大屏科技蓝
高饱和渐变
玻璃拟态
厚重阴影
后台管理模板侧栏
```

### 8.4 响应式

桌面端地图撑满内容区，当前五指标一行展示，详细数据页图表两列布局。

平板端地图高度降低，五指标改成 3 + 2，详细数据页必要时改成单列。

手机端地图仍保留站点点位，但标签缩短为“站名 + 温度”。五指标改为两列或单列，详细数据页图表全部单列。

## 9. 中国地图合规

主页使用中国轮廓图。地图资源必须来自官方标准地图或经过审查的地图文件。

建议来源：

```text
自然资源部标准地图服务
https://bzdt.ch.mnr.gov.cn/
```

处理规则：

```text
只使用官方地图资源。
只保留中国轮廓和必要边界。
不使用来源不明的 GeoJSON。
不自行绘制或修改国界线。
地图文件在报告中注明来源、版本和审图号。
```

站点点位只表示城市位置，用于切换和总览，不提供 GIS 查询能力。

## 10. 运行与演示

演示前先导入历史数据：

```text
1. 拉取唐山、北京、上海的 Open-Meteo 历史数据。
2. 转成课程兼容 CSV。
3. 上传 HDFS `/weathertextdb`。
4. 创建 Hive 外部表。
5. 运行 Spark 分析任务。
```

演示时启动服务：

```text
1. 启动 HDFS、YARN、HBase、HBase Thrift、Hive。
2. 启动采集服务。
3. ESP32 上电并连接 Wi-Fi。
4. 启动 FastAPI。
5. 启动 Next.js 前端。
```

推荐演示顺序：

```text
1. 打开主页，展示地图和三个站点天气。
2. 点击北京、唐山、上海，观察五指标读数切换。
3. 让 ESP32 采集唐山温湿度，展示硬件链路。
4. 进入详细数据页，展示趋势、统计和数据记录。
5. 用 HDFS、HBase、Hive、Spark 的命令或截图补充说明后台链路。
```

## 11. 验收标准

数据链路应满足：

```text
ESP32 能发送唐山温湿度。
采集服务能合并 Open-Meteo 当前气象数据。
HBase 能查到实时记录。
HDFS 能查到历史记录和实时追加记录。
Hive 能查询 `weather_table`。
Spark 能生成窗口均值和摘要结果。
```

前端应满足：

```text
主页显示合规中国轮廓图。
北京、唐山、上海三个站点可点击。
地图点位显示天气状态和温度。
下方五指标框随站点切换。
详细数据页显示趋势、统计摘要和记录。
移动端内容不重叠。
```

报告应说明：

```text
数据来源。
FastAPI 替换 MTAS 的原因。
Next.js、shadcn/ui、Recharts 替换原可视化方案的原因。
Open-Meteo 替代随机仿真的原因。
HBase、HDFS、Hive、Spark 在系统中的职责。
地图来源和合规处理。
```
